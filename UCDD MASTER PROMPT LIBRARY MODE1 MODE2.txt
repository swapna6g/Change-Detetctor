================================================
HOW TO USE THIS FILE
================================================

This file contains all AI prompts for UCDD population
and validation. It is organized into sections — one per
processing task or column group.

HACKATHON IMPLEMENTATION GUIDE:

  STEP 1 — Set the processing mode in your code:
    processing_mode = "Mode 1"   (default — use this)
    processing_mode = "Mode 2"   (optional — plug in if needed)

  STEP 2 — For every API call to Claude:
    Always inject Section 0 (System Context) first.
    Then inject the relevant section for your task.
    Fill in all {{placeholder}} values with real data.

  STEP 3 — Mode 2 activation:
    To enable Mode 2: activate Section 1 in your pipeline.
    To disable Mode 2: skip Section 1 entirely.
    No other changes needed anywhere in this file.

  STEP 4 — Process columns in this order per UCDE:
    Metadata (once per session) → Col 1 → Col 2 →
    Col 3 → Col 4 → Col 5 → Col 6+7+8 → Col 9+10 →
    Col 11+12 → Change Matrix row assembly

VARIABLE REFERENCE:
  {{processing_mode}}      Mode 1 or Mode 2
  {{ucde_name}}            Current Use Case Element Name
  {{ucde_desc}}            Current Use Case Element Description
  {{source_evidence}}      Extracted text from report
  {{existing_value}}       Current value in existing UCDD (Mode 1)
  {{disposition}}          Add/Update/Delete/No Change/Review Required
  {{data_type}}            numeric count/rate/date/text/calculated/identifier
  {{data_source_type}}     approved ADS/third-party/non-application/external
  {{is_calculated}}        true or false
  {{report_type}}          regulatory or internal
  {{source_section}}       Section reference from report
  {{uc_type}}              external or internal
  {{admf_identifier}}      Confirmed Use Case Identifier from ADMF

ACTIVE MODE: Mode 1 — Change Detection

You are performing semantic comparison between two versions
of the same use case report. Your task is to identify every
meaningful change between the prior approved version and the
newly submitted version.

PRIOR APPROVED REPORT — Section: {{section_name}}
{{prior_report_section_content}}

NEW SUBMITTED REPORT — Section: {{section_name}}
{{new_report_section_content}}

COMPARISON RULES:
1. Compare sections with the same semantic meaning even if
   headings differ between versions
2. Detect changes in:
   - Data element names or definitions
   - Calculation logic or aggregation rules
   - Business rules or inclusion/exclusion criteria
   - Report structure or section organization
   - Data sources referenced
   - Scope of reporting period or population
3. Do NOT flag placeholder changes:
   ## to ### or ### to ## are NOT meaningful changes
   Normalize all ## patterns to [REDACTED] before comparing
4. Distinguish between:
   - Wording changes that change meaning (flag as change)
   - Wording changes that preserve meaning (flag as no change)
5. For each detected change identify:
   - What it was in the prior version
   - What it is in the new version
   - Which section of the report it comes from
   - Whether it is a structural or semantic change

EARLY EXIT RULE:
If no meaningful changes are detected across all sections:
  Return early_exit: true
  All UCDEs will be assigned No Change disposition
  Skip Sections 3, 4, 5, and 6 in this pipeline call

You are mapping detected report changes to specific UCDEs

DETECTED CHANGES FROM SECTION 2:
{{changes_from_section_2}}

EXISTING UCDD ELEMENTS:
{{existing_ucde_list}}

MAPPING RULES:
1. For each detected change identify which UCDE or UCDEs
   in the existing UCDD are affected
2. A change in calculation logic affects the UCDE whose
   description or DQ checks reference that calculation
3. A change in element name in the report maps to Update
   disposition for that UCDE
4. A new element in the report with no matching UCDE maps
   to Add disposition
5. An element in the existing UCDD not found in the new

   maps to No Change disposition
7. When the impact is unclear or ambiguous assign
   Review Required disposition with specific reason
8. Every UCDE in the existing UCDD must receive exactly
   one disposition — no UCDE should be skipped
