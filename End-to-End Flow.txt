Phase 0 — System Startup (Before Any User Interaction)
0.1 — Load Configuration
Backend reads system_config.json into memory.
Reads mode_2_enabled to determine whether to show mode toggle on UI.
Reads all threshold values, batch size, accepted formats.
0.2 — Load Prompt Library
Backend reads UCDD_MASTER_PROMPT_LIBRARY.txt into memory.
Parses it into sections using === SECTION N === markers.
Each section stored as a named entry for fast retrieval.
0.3 — Load Guideline Rules
Backend reads guideline_rules.txt into memory.
Parses into sections by column name markers.
Rules engine stores these for validation use throughout processing.
0.4 — Load Routing Map
Backend reads prompt_routing_map.json into memory.
Maps step names to section numbers for both modes.
0.5 — Load ADMF Reference
Backend reads admf_identifier_lookup.json into memory.
Stores valid Use Case Identifiers and their corresponding names.
0.6 — Load DQ Validation Reference
Backend reads dq_check_validation.json into memory.
Stores valid check numbers for Accuracy, Completeness, Timeliness.
Rules engine uses this for hard validation — no AI needed.
0.7 — Verify Templates Exist
Backend confirms all three Excel templates are present:
CHANGE_MATRIX_TEMPLATE.xlsx
UCDD_OUTPUT_TEMPLATE.xlsx
AUDIT_LOG_TEMPLATE.xlsx
If any missing — log error and alert on startup.
0.8 — Verify API Key
Backend reads ANTHROPIC_API_KEY from .env file.
Sends a test ping to confirm the key is valid.
If invalid — log error and display alert to user.

Phase 4 — Document Processing Layer
4.1 — Parse New Use Case Report
Backend detects file format from extension.
For .pdf files:
Extract text layer using PDF text extraction.
If extraction returns empty — flag as scanned PDF.
Scanned PDFs require OCR — for hackathon flag as unsupported and alert user.
For .docx or .doc files:
Extract text preserving headings, bullets, paragraphs.
Convert heading styles to markdown markers.
For .pptx files:
Extract text from each slide in order.
Preserve slide title as section heading.
Extract text from all text boxes on each slide.
Extract table content from slides — preserve row/column structure.
Note: PPTX is a critical format for this use case because both sample reports are PowerPoint presentations.
For .txt files:
Read directly — clean encoding artifacts.
Output: raw extracted text with preserved structure.
4.2 — Parse Prior Approved Report (Mode 1 only)
Same process as 4.1 applied to the prior report file.
Output: raw extracted text with preserved structure.
4.3 — Parse Existing UCDD (Mode 1 only)
For .xlsx or .xls:
Read Sheet 1 — Use Case Dictionary tab.
Read Sheet 2 — Upstream PDE Sourcing tab.
Convert each row to a structured object with exact column names as keys.
Extract Use Case Identifier and Use Case Name from header rows.
For .csv:
Read as tabular data.
Map columns to expected UCDD column names.
Flag any missing expected columns.
Output: structured list of existing UCDE objects.
4.4 — Placeholder Detection
Scan all extracted text from all documents.
Find all occurrences of ## or ### or #### patterns.
Replace with [REDACTED] token.
Log count of replacements per document.
This prevents placeholder values being treated as real data.
4.5 — Structure Documents Using AI
For each report call Claude with Section 0 plus structuring prompt.
Prompt asks Claude to organize content into labeled sections:
Background and Purpose
Data Elements
Calculations and Business Logic
Report Outputs
Data Sources
Business Rules and Governance Notes
Output: structured JSON with section labels as keys and extracted content as values.
4.6 — Generate Section Structure Map
Claude reads the structured content and produces section hierarchy:
Top-level sections with exact titles.
Sub-sections under each top-level section.
Hierarchy Level 1 and Level 2 candidates for each section.
This replaces the manually pre-built structure map discussed earlier.
4.7 — Normalize Terminology
Claude compares terminology across both reports.
Identifies synonyms or variations of the same concept.
Produces a normalization map — old term maps to new term.
Applied before comparison to ensure semantic accuracy.
4.8 — Validate Parsing Success
Check that Data Elements section is not empty in either report.
Check that UCDD has at least one UCDE row.
Check that Use Case Identifier in UCDD matches ADMF confirmed value.
If any check fails — stop processing — update session status to "failed" — alert user with specific reason.
Update session status to "parsed" — progress tracker shows Stage 1 and 2 complete.

Claude confirms match or flags mismatch.
If mismatch — flag for human review — do not block processing.

Write confirmed Use Case Name to output UCDD template header.

Phase 6 — Stage 2 — Semantic Comparison (Mode 1) or Element Discovery (Mode 2)
MODE 1 PATH:
6.1 — Early Exit Check
Calculate semantic similarity score between new and prior report.
If similarity above early_exit_similarity_threshold (0.95):
Assign No Change to all existing UCDEs.
Skip Steps 6.2 through 6.4.
Jump directly to Phase 8.
Update session status — log early exit reason.
6.2 — Change Detection Engine
Load Section 0 and Section 2 from prompt library using routing map.
Send structured new report and prior report section by section.
Claude identifies every meaningful change per section.
Returns structured list of detected changes with source references.
Update session status — Stage 3 In Progress.
6.3 — Impact Mapping Engine
Load Section 0 and Section 3 from prompt library.
Send detected changes from 6.2 plus existing UCDE list.
Claude maps each change to affected UCDEs.
Assigns preliminary disposition to each UCDE.
Returns UCDE disposition list.
Update session status — Stage 4 In Progress.
6.4 — Multi-Dimensional Detection
Scan detected changes and UCDE list for multi-dimensional elements.
Elements appearing across country, segment, or time dimensions flagged.
For each flagged element:
Load Section 14 from prompt library.
Claude recommends Approach 1 or Approach 2.
Both approaches previewed and stored for reviewer decision.
These are not auto-resolved — presented to reviewer in dashboard.
MODE 2 PATH:
6.5 — Element Discovery Engine
Load Section 0 and Section 1 from prompt library.
Send structured new report content to Claude.
Claude discovers all distinct data elements.
Returns complete element list with data types, sections, hierarchy candidates.
Multi-dimensional detection runs as part of discovery.
Update session status — Stage 3 complete.

Phase 7 — Stage 3 and 4 — Disposition and Materiality
7.1 — Batch UCDEs for Processing
Take complete UCDE list — either from existing UCDD (Mode 1) or discovery (Mode 2).
Split into batches of batch_size from config (default 50).
For your demo use case with 6 UCDEs — one batch only.
7.2 — Column-by-Column Processing Per UCDE
For each UCDE in the batch process all 12 columns in sequence.
Each column uses its section from the prompt library via routing map.
Column 1 — Section 5:
Generate or validate Use Case Element Name.
Apply naming standards from guideline_rules.txt.
For No Change in Mode 1 — carry forward existing value.
Column 2 — Section 6:
Generate or validate Use Case Element Description.
Apply description standards.
For No Change in Mode 1 — carry forward existing value.
Columns 3 and 4 — Section 7:
Determine Hierarchy Level 1 and Level 2.
Use section structure map from Phase 4.6.
For No Change in Mode 1 — carry forward existing values.
Column 5 — Section 8:
Extract MDRM code if visible in report.
Do not fabricate — flag if uncertain.
For No Change in Mode 1 — carry forward existing value.
Columns 6, 7, 8 — Section 9:
Select Accuracy Dimension.
If Multiple — populate Accuracy Dimension MULTI field.
If N/A — generate justification meeting all 5 quality criteria.
For No Change in Mode 1 — carry forward all three columns.
Columns 9, 10 — Section 10:
Select Completeness Dimension.
If N/A — generate justification.
For No Change in Mode 1 — carry forward both columns.
Columns 11, 12 — Section 11:
Select Timeliness Dimension — only Data Delivery Check or N/A.
If N/A — generate justification.
For No Change in Mode 1 — carry forward both columns.
7.3 — Rules Engine Validation Per Column
After each column is returned by Claude the rules engine validates:
For text fields:
Character count within limit.
Does not start with "This" (description).
No programming language detected.
For dropdown fields:
Value is from valid dropdown list using DQ validation reference JSON.
If Multiple selected — MULTI field is populated.
If MULTI field populated — all check numbers are valid (1-12).
If N/A selected — justification field is not empty.
For justification fields:
Score against 5 quality criteria.
If score below review_required_threshold (3) — flag as Review Required.
Record which criteria failed.
For MDRM field:
Character count 8 or under.
No fabricated values — only accept if Claude confirms source.
7.4 — Confidence Score Calculation
For each UCDE calculate confidence score:
Semantic similarity delta — 40% weight.
Number of sections impacted — 30% weight.
Sensitivity of affected section — 30% weight.
Final score 0 to 100.
If score below confidence_threshold (55) — flag as Review Required.
7.5 — Materiality Assessment
For Add, Update, Delete, and Review Required UCDEs only:
Claude assesses whether change is material.
Returns Y or N flag plus plain language rationale.
Returns source evidence — exact section and passage reference.
No Change UCDEs — skip materiality — assign N automatically.
7.6 — Assemble Change Matrix Row
Load Section 12 from prompt library.
Combine all column outputs for this UCDE.
Produce one complete Change Matrix row:
UCDE Name, Disposition, Change Summary, Material Y/N,
Confidence Score, Materiality Rationale, Source Evidence,
Policy Compliance Status, Compliance Flags, Review Required.
7.7 — Write UCDE Row to Output Files
Write completed UCDE row to session UCDD output file.
Write Change Matrix row to session Change Matrix file.
Write column-level detail to Audit Log file — one row per column.
Update session status — Stage 5 and 6 In Progress.

Gaps Identified and Addressed
Here are the items that were missing or unclear before this audit:
PPTX format now included in all upload zones and parsing steps. This was missing in earlier architecture discussions and is critical because both sample reports are PowerPoint files.
Scanned PDF handling — flagged as unsupported for hackathon. If reports are image-based PDFs text extraction will fail. User should be warned to upload text-based PDFs or convert scanned PDFs first.
Session history — not previously planned but now part of the dashboard. Allows multiple runs without overwriting.
Multi-dimensional decision — now correctly placed in the dashboard as a human decision point — not auto-resolved by AI.
Early exit flow — fully mapped. If triggered it skips Phases 6 and 7 and jumps directly to Phase 8 with all No Change dispositions.
Reject disposition — added to reviewer actions. Previously only Accept and Modify were planned. Reject is important for cases where the AI generated a completely wrong row.
Processing page with named stages — prevents the blank screen problem during AI processing. Judges see the pipeline running live.
Metadata confirmation — ADMF identifier confirmed before any column processing begins. Previously this was mentioned but not placed correctly in the flow.
Download before approval — WIP files downloadable at any time. Previously only final files were mentioned. Reviewers often want to review in Excel before completing the dashboard workflow.
